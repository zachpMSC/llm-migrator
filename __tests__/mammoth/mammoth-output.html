1. Purpose. The purpose of this procedure is to establish requirements for software and digital data systems used in medical device sterilization activities. This includes all Medical Device Parameter Review Audits. This procedure is intended to address the applicable elements of US 21 CFR Part 11 and other associated guidance such as ISPE GAMP 5 (Risk-Based Approach to Compliant GxP Computerized Systems).
  2. Scope. New or revised software and digital data used in medical device sterilization, quality control, complaint handling and other product related activities must be validated per this SOP.
  3. Definitions
    1. Commercial Off-the-Shelf Software (COTS): Software developed externally in which the company does not control or revise the software specification. This type of software is validated by the supplier and is only verified for functional usage at MSC. Verification means confirmation by examination and provision of objective evidence that specified functional and/or operational requirements have been fulfilled.
    2. Software Validation Team: Engineering, Quality Assurance and Information Technology representatives who review software for validation status and validation requirements.
    3. Firmware: lower level software embedded into read only memory of a hardware component. Many equipment and monitoring systems are setup using manufacturer installed firmware. These do not have user applications that are intended for more complex and variable operations that are associated with higher level software programs, but are set to only accept specific input types that then provide defined outputs. Firmware cannot be updated by the user and any updates are provided by the hardware manufacturer. Firmware based equipment or instruments will only require operational verifications (see definition above) and are not software validated by the user. Below is a typical outline of the hardware-firmware-software hierarchy.

Application software – a Quality Management tool such as MasterControl or ERP (Enterprise Resource Planning) software like SAP or Oracle.

Middleware – software used to process data for exporting to other systems.

Operating systems (such as Windows) – User Interfaces, software drivers, OS services that exchange information within a program system that allows it to operate as intended.

Firmware – low level software imbedded into a hardware non-volatile memory. It can only be changed or modified by the original hardware manufacturer.

Hardware – the physical components of the computer or equipment / instrument.

    1. Data Integrity Management: the process via which applicable data are confirmed as complete, consistent, and accurate throughout the data lifecycle – from initial data generation, processing, transfer, archiving, and destruction. The ALCOA principles (Attributable, Legible, Contemporaneous, Original, and Accurate) are the key elements applicable to Data Integrity for GxP records. Data Integrity considerations apply to all digital data generated that directly supports process or product Quality requirements at MSC.
  1. Responsibilities / Authority
    1. Quality Assurance Department: is responsible for verifying new and upgraded application software / computer systems that support medical device sterilization, quality control, complaint handling and other product related activities and authorized to release final software based on successful completion of validation / verification testing.
    2. IT Department: When software intended for direct application for process cycle or quality assurance applications is developed outside of the company, the developer must ensure that the validation requirements of the software have been met. Responsible for validating all computer systems utilized in any manufacturing and test functions.
    3. Management: is responsible for establishing IT business priorities and ensuring there are adequate resources to conduct software validations.
    4. Software Validation Master Plan considerations will be made as part of an overall Validation Master Plan process for use in identifying and managing software / digital systems per the following provisions:

Includes an overview of each process, the validation or verification approach and the rationale for following it, applying the elements listed as applicable for the system being used:

  • Design Qualification (DQ): Identifies key functionality, design specifications and software requirements. For customized Off the Shelf (OTS) and in house developed software only.
  • Risk Assessment and Management: Documents the risk factors and mitigations associated with using the software to process medical devices.
  • Software Categorization: per the identifications found in this SOP per the Definitions and Procedure categories.
  • Vendor Qualification (VQ) – if using commercially available OTS software, is the software supplier already qualified, or does a supplier evaluation need to be conducted?
  • Quality, Operations, and IT Resources: Define the roles and responsibilities in the overall validation plan process, including maintaining consistency with GxP or GMP and managing any relevant Standards of Practice (SOPs).
  • Hardware Specifications: Lists all hardware and software requirements.
  • Installation Qualification (IQ) for Install: Confirms that all hardware, software and networks have been installed and configured properly. 
  • Installation Qualification (IQ) for Upgrades: Confirms that all hardware, software and networks will be installed properly after upgrades are performed.
  • Operational Qualification (OQ): Verifies that the software is functional; key features will perform as expected; all necessary modules can be accessed; and that data can be viewed, uploaded and exported.
  • Performance Qualification (PQ): Lists the tests and results that confirm the software will perform as needed to produce the desired goods in compliance with FDA guidelines.
  • Support and Maintenance: Specifies who will be responsible for ongoing maintenance and support, as well as what their responsibilities are.
  • SOP and Change Control: Lists procedures subject to change that could result from the Validation Plan.

The Validation Master Plan review will normally be developed in Q1 of each calendar year, although it may be developed or updated at other times as necessary to account for changing quality or business conditions.

  1. Procedure
    1. COTS software requiring validation / verification may not be used until validation or verification of the software is complete. This includes software revisions of previously validated software unless determined to not require validation by the quality assurance department.

Evaluations regarding whether a software validation, verification, or basic Data Integrity assessment only is required will be done by the Software Validation Team. Potential exemptions from validation, verification, or Data Integrity assessments should be evaluated per GAMP 5 risk assessment guidelines:

Category 1: Infrastructure Software (includes operating systems, database managers, middleware, and any other tools for network monitoring, batch job scheduling tools

etc.) – generally subject to software validation if related to product or process Quality.

Category 2: N/A – see note below

Category 3: Non-Configured Software (includes all COTS software that comes with a default configuration) – generally applicable only as software verification category for use in direct process or product Quality applications. MSC would not perform software validations on non-custom configured commercial software (Excel, etc).

Category 4: Configured Software (includes COTS systems that can be configured as per business processes as well as customized) – generally subject to software validation, dependent on customizations.

Category 5: Custom Software – internally or 3rd party contracted software designed and developed specifically site operational or Quality systems. Software validation required if related to product or process Quality.

Note: “Firmware” was earlier classified as “Category 2” under GAMP 4. This classification was removed under GAMP 5 and is generally not subject to software validation considerations.

Software system risk considerations and process flow:

      1. Exemptions to validations: Any software that is not used in medical device sterilization or other product related quality or compliance activities as described below is exempt from this procedure.
      2. Examples of software exemptions include:
        1. Software whose use is incidental to a regulated activity (i.e. browsers, utilities, virus checkers, media players, email programs, etc.)
        2. Software used to facilitate the print a hard copy of a document that may support a regulated activity but where the hard copy serves as the official document (i.e. printing a purchase order, Microsoft word, Adobe illustrator, font software, etc.).
        3. Software used to support finance, sales functions, etc. that are not used for vendor management or other compliance purposes.
        4. Software used to support research activity.
        5. Software that produces results verified through an independent review in addition to a validated process.
        6. Operating Systems and related network operating software shall be validated as part of an overall validation effort, as applicable.
    1. Required Validation Effort: Software is categorized as: Exempt, Minor, and Major.
      1. Exempt from Validation: Software and PLC packages that are exempt will be noted on either the PLC or Software change form as exempt and a rationale for exemption will be noted. Software is approved for intended use. Note that even for exempt processes a Data Integrity evaluation should still be made. For example, if there is a digital system that is used to create hardcopy records that then become the official Quality record, the original electronic data may still be subject to Data Integrity considerations if that data is kept in electronic storage.
      2. Minor: A justification for use or an executed minor validation or verification must be documented prior to use on either the Software or PLC change form.
        1. The justification should include:
          • Description of change
          • Reason for change
          • If customer notification is required
        2. Minor validation can include the following and should be documented like the justification:
          • Simple testing on the overall functionality of the software, including screen shots.
      3. Major: Software installation or change will require a Quality Plan or CAPA. The following steps are required:
        1. Quality Plan or CAPA
        2. Develop Software Protocol
        3. Approve Protocol
        4. Complete Validation / Verification
        5. Approve Validation / Verification

The extent of any required testing will be determined by the results of the risk analysis. High risk software should be tested more extensively than software that poses a moderate risk.

d. Parameter Review Audits: To check for conformity to a customer cycle. No Quality Plan is required. A software change form is required.

  1. Parameter Review Audits will be accepted as validated when:
    • A Parameter Review Audit has been printed which displays in-parameter values for each area of the Parameter Review Audit being validated and shows no deviations.
    • The Parameter Review Audit showing no deviations has been verified by a member of Quality or Management and has been signed / dated as such.
    • A Parameter Review Audit has been printed which displays low-parameter deviations for each area of the Parameter Review being validated. Also, the deviation value listed in the deviation column must be correct.
    • A Parameter Review Audit has been printed which displays high-parameter deviations for each area of the Parameter Review being validated. Also, the deviation value listed in the deviation column must be correct.

  1. Validation of Excel spreadsheets used in calibration activities.
    1. Spreadsheets can be used to record data and as such need to be managed to ensure the continued integrity and security of that data. At MSC Excel Spreadsheets used to conduct formula calculations fall into GAMP Category 3 guidelines and do not require full validation activities but should be verified for functional accuracy and data security.
    2. The Operational Qualification shall be written to document the instructions for the testing. The Operation Qualification should include the following tests:
      1. Calculation Test to verify that correct formulas are entered.
      2. Range/Precision Test to verify calculations are accurate when extremely large or small values are entered, and will include “Challenge condition” testing for the application (e.g., invalid inputs, stress testing).
      3. Acceptance criteria shall be established for each test and written into the Operation Qualification Protocol.
        1. Acceptance Criteria - Calculated values from the spreadsheet application must be identical to those obtained from a calculator (preferably with printout capabilities).
        2. Range/Precision Test - These are performed to verify calculations will still be accurate when extremely large or small values are entered. Enter extremely large values at the correct precision (e.g. enter 99.99 for 0.01 precision). For aggregate functions, enter 0 in one or more fields with non-zero values in other fields. Duplicate the formula calculations using a calculator and compare the results/printouts. Repeat this test with extremely small values, and report the results. Also conduct tests using invalid inputs and large data sets (data 3 times greater than normal expected data sets) to ensure that either applicable error messages are noted or for larger data sets the formulas still calculate accurately.
        3. Acceptance Criteria - Calculated values from the spreadsheet application must be identical to those calculated manually using a calculator (with correct precision and consistent with agreed Rounding principle).
    3. Re-qualification of Spreadsheet Application Formulas
      1. Spreadsheet applications used for calibration applications will be reviewed / verified every 3 years to address the validity of the validation documentation which includes formula integrity. Reminder is in task management system.
      2. Re-qualification may also be triggered by the installation of an upgrade of the spreadsheet program (Change Control System), or by changes in the network file system.


  1. Record Requirements. The completed validation reports should be filed with the validation package by the quality department. Validation and Verification protocol activities should be developed and approved using the protocol template MSC-Form-0189. This template can be modified as necessary to account for the type of software category and the nature of the change (Minor or Major).
